# Fork-Sync Efficiency Improvements

**Date:** 2026-02-17
**Branch:** feature/fork-sync-v2 (to be created)
**Status:** Approved

---

## Problem Statement

The current fork-sync workflow has two critical efficiency problems observed in a real session (2026-02-17):

1. **Fragility**: A parallel git checkout mid-merge silently aborted in-progress conflict resolution, forcing a full redo of 928 conflict resolutions.
2. **Speed**: 928 conflicts — ~880 of them purely mechanical (acceptWithRebrand) — are resolved manually by Claude one by one, which is slow and error-prone.

Secondary problem: evaluation state (Phase 1.6) goes stale whenever mirror advances between evaluation and merge, requiring a manual staleness decision each time.

---

## Goals

- **Interrupt-proof**: Parallel git sessions cannot destroy in-progress merge work.
- **Fast**: The 880+ mechanical conflicts resolve in seconds via a deterministic script, not by Claude staging files one at a time.
- **Idempotent**: If anything fails, recovery is `rm -rf worktree`, recreate, re-run script — ~30 seconds.
- **State-fresh**: Evaluation state auto-updates when mirror advances incrementally, preserving completed module decisions.

---

## Design

### Phase 1.7 — Generate Resolution Script

**Trigger:** Immediately after Phase 1.6 evaluation is complete (shopping list finalized).

**Output:** `.claude/skills/fork-sync/scripts/resolve-conflicts.sh`

The script is generated from `evaluation.json`'s `mergeShoppingList` and encodes every resolution decision:

```bash
#!/bin/bash
# Auto-generated by fork-sync Phase 1.7
# Generated: 2026-02-17T00:00:00Z
# Mirror head: 2cf82c357
# Shopping list: 8 keepOurs, 3 manualMerge, 880 acceptWithRebrand,
#                32 extensionPackageJson, 72 acceptDeletion, 3540 autoAccept

set -e
REBRAND='s/from '"'"'openclaw'"'"'/from '"'"'@nikolasp98\/minion'"'"'/g'

resolve() {
  local f="$1" strategy="$2"
  case "$strategy" in
    ours)       git checkout --ours "$f" 2>/dev/null && git add "$f" && echo "✓ ours:    $f" ;;
    theirs)     git checkout --theirs "$f" 2>/dev/null && git add "$f" && echo "✓ theirs:  $f" ;;
    rebrand)    git checkout --theirs "$f" 2>/dev/null
                sed -i "$REBRAND" "$f"
                git add "$f" && echo "✓ rebrand: $f" ;;
    extpkg)     git checkout --theirs "$f" 2>/dev/null
                sed -i 's/"openclaw": "workspace:\*"/"@nikolasp98\/minion": "workspace:*"/g' "$f"
                git add "$f" && echo "✓ extpkg:  $f" ;;
    rm)         git rm -f "$f" 2>/dev/null && echo "✓ removed: $f" ;;
  esac
}

# --- keepOurs (8 files) ---
resolve "Dockerfile" ours
resolve "docker-compose.yml" ours
# ... (full list from shopping list)

# --- acceptWithRebrand (880 files) ---
resolve "src/agents/minion-tools.ts" rebrand
# ... (full list)

# --- extensionPackageJson (32 files) ---
resolve "extensions/bluebubbles/package.json" extpkg
# ... (full list)

# --- acceptDeletion (72 files) ---
resolve ".agents/AGENT_SUBMISSION_CONTROL_POLICY.md" rm
# ... (full list)

# --- autoAccept (theirs, no rebrand) ---
resolve "docs/channels/telegram.md" theirs
# ... (full list)

# --- MANUAL MERGE REQUIRED (3 files) ---
# These are NOT resolved by this script. Claude must handle them.
# package.json:
#   Accept upstream deps, keep name=@nikolasp98/minion, keep version bump
# .github/workflows/ci.yml:
#   Accept upstream CI improvements, keep fork branch config + MINION vars
# .github/workflows/docker-release.yml:
#   Accept upstream, keep DEV/main branch triggers

echo ""
echo "✓ Auto-resolution complete."
echo "MANUAL: Resolve these 3 files then run: git add . && git commit"
echo "  - package.json"
echo "  - .github/workflows/ci.yml"
echo "  - .github/workflows/docker-release.yml"
```

**Properties:**

- **Idempotent**: Files already staged are silently skipped
- **Logged**: Each resolution prints `✓ <strategy>: <file>`
- **Committed**: Script is saved to the skills directory and committed to git as part of the Phase 1.7 step, surviving session restarts

### Phase 2 — Worktree-Based Merge (replaces direct DEV merge)

Phase 2 **always** uses a git worktree. The direct-merge-on-DEV approach is retired.

**Workflow:**

```bash
# Step 1: Create isolated worktree pointing at DEV
git worktree add ../openclaw-merge-worktree DEV

# Step 2: Merge mirror into worktree (conflicts land in worktree, not main workspace)
git -C ../openclaw-merge-worktree merge mirror --no-commit --no-ff

# Step 3: Run resolution script (resolves ~925 of 928 conflicts in seconds)
bash .claude/skills/fork-sync/scripts/resolve-conflicts.sh

# Step 4: Claude resolves the 3 manualMerge files
#   package.json, ci.yml, docker-release.yml

# Step 5: Commit the merge
git -C ../openclaw-merge-worktree commit -m "Merge upstream changes from mirror"

# Step 6: Push DEV from worktree
git -C ../openclaw-merge-worktree push origin DEV

# Step 7: Cleanup
git worktree remove ../openclaw-merge-worktree
```

**Why this prevents interruption:**
The main workspace is on `feature/dashboard-revamp` (or wherever the developer is working). Git operations in the main workspace — commits, checkouts, stash pops — have zero effect on the worktree. The merge state lives entirely in `../openclaw-merge-worktree/.git/MERGE_HEAD`.

**Recovery path if interrupted:**

```bash
git worktree remove --force ../openclaw-merge-worktree
# Recreate and retry — full reset in ~30 seconds:
git worktree add ../openclaw-merge-worktree DEV
git -C ../openclaw-merge-worktree merge mirror --no-commit --no-ff
bash .claude/skills/fork-sync/scripts/resolve-conflicts.sh
# ... then manualMerge files + commit
```

### Evaluation State Delta Updates

When `fork-sync` is invoked and `evaluation.json` exists with a stale `snapshot.mirrorHead`:

**Old behavior:** Warning → user decides to re-evaluate or skip → often requires full re-evaluation.

**New behavior:** Automatic delta update:

1. Compute delta: `git log --oneline <old_mirror_head>..<new_mirror_head>`
2. For each new commit, apply Tier 1/2/3 auto-categorization heuristics
3. For each affected file:
   - If the file belongs to an already-completed module → inherit that module's decision
   - If the file is in `forkFeatureIndex` → flag for manual review
   - Otherwise → add to `autoAccept` or `acceptWithRebrand` per path rules
4. Append new entries to `mergeShoppingList` categories
5. Update `snapshot.mirrorHead` to current mirror HEAD
6. Re-generate the resolution script (Phase 1.7) to include new files

**Example (today's session):** 107 new commits, mostly iOS share extension, test fixes, docs, voice-call improvements. Delta update would have:

- Added ~40 new `autoAccept` entries (iOS, docs)
- Added ~60 new `acceptWithRebrand` entries (test files, src/ changes)
- Added ~7 new `extensionPackageJson` entries (voice-call, bluebubbles)
- Zero new `manualMerge` entries
- Total time: ~30 seconds, zero human interaction needed
